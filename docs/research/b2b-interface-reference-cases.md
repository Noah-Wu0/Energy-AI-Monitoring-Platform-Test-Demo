# B 端界面参考案例

## 调研结论

本项目不适合参考营销型 SaaS 首页，也不适合做成纯炫酷大屏。更合适的参考方向是工业 IoT、能源管理、SCADA、跨地点运营和审批/审计工作台。

这些案例的共同点：

1. 面向真实工作流，而不是只展示指标。
2. 支持从概览下钻到事件、证据和动作。
3. 重视状态层级、异常突出、角色视图和审计留痕。
4. 图表服务判断，不做装饰性图表。
5. 领导层与操作层信息密度不同，需要分层展示。

## 案例 1：美国能源部能源管理 Dashboard 指南

链接：https://www.energy.gov/eere/femp/articles/general-recommendations-federal-data-center-energy-management-dashboard-display

参考价值：

- 官方能源管理类 dashboard 参考。
- 强调 dashboard 应围绕目标和利益相关者设计。
- 强调实时或准实时监控、趋势展示和可行动数据。

对本项目启发：

- 监控态势页不要只放 KPI，要明确“看完以后下一步做什么”。
- 能耗曲线、预测带、异常说明需要服务监管动作。
- 领导层和技术部门应看到不同层次的信息。

## 案例 2：DemandQ 能源成本管理 Dashboard

链接：https://technext.it/portfolio/energy-cost-management-dashboard-ui-ux-redesign/

参考价值：

- 典型能源管理 B 端数据密集界面。
- 其改版重点是从表格堆叠转向图表、视觉指标和结构化布局。
- 强调在单页放多种数据图表但不压垮用户。

对本项目启发：

- 能耗异常页要以曲线图为主，表格只作为证据补充。
- 状态色和视觉指标需要稳定，不要每页换一套。
- 复杂信息需要分区：状态、趋势、解释、证据、动作。

## 案例 3：CABB SCADA 工业 UX 改版

链接：https://jorgenrique.com/cabb/

参考价值：

- 工业/政府/内部工具场景，和能源部监管 demo 接近。
- 改版重点不是视觉炫技，而是导航清晰、数据可读、降低认知负担、防止错误。
- 案例强调桌面优先、24/7 运行、真实操作员工作流。

对本项目启发：

- 人工复核页必须有确认模式、理由填写和错误预防。
- 审计链页面要能快速看出“谁做了什么”，不要做成漂亮但难读的时间轴。
- 对值班/监管用户，少一些解释性废话，多给直接状态和动作。

## 案例 4：多地点运营企业 Dashboard

链接：https://stellarcode.io/case-study/enterprise-dashboard-multi-location-operations/

参考价值：

- 强调统一运营视图、工作流状态、审批自动化、角色权限和审计状态。
- 技术上采用 React、Node、PostgreSQL、Docker 等常见企业栈。

对本项目启发：

- 我们的 demo 也应把“事件状态”和“人工复核状态”作为核心对象。
- 未来如果扩展多个作业区，可以从单事件流程扩成多地点监管视图。
- 审计留痕不是附属页面，而是工作流本身的一部分。

## 案例 5：工业 IoT 工厂数字化 Dashboard

链接：https://ingenios.io/case-studies/factory-digitalization

参考价值：

- 从手工记录和表格转向自动数据采集、实时报警、历史分析和报告导出。
- 同时服务操作员、经理和管理层。

对本项目启发：

- 我们应明确三层视图：态势概览、事件处置、报告汇总。
- 报告生成页不能只是一个漂亮卡片，要体现数据快照和审计引用。
- “实时报警 + 历史趋势 + 定期报告”可以成为 demo 的稳定叙事结构。

## 推荐参考组合

本项目建议采用以下参考组合：

1. 能源部指南作为信息层级和能源管理逻辑参考。
2. CABB SCADA 作为工业操作体验和错误预防参考。
3. DemandQ 作为能源数据可视化参考。
4. 多地点运营 Dashboard 作为工作流和审计状态参考。

## 不建议参考的方向

- 过度科幻的深色大屏。
- 单纯 KPI 卡片墙。
- 只有地图和闪烁点位，没有事件处置流程。
- 面向销售展示的通用 SaaS dashboard。
- 图表很多但没有清晰动作入口的分析页面。
