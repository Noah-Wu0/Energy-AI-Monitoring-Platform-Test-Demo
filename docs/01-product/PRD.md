# 阿克套油田能源作业区 AI 监管闭环 Demo PRD

更新时间：2026 年 5 月 18 日

## 1. 当前判断

本项目当前不应先拆成多人并行开发工程，也不应直接追求完整可点击系统。现阶段的核心任务是把能源部演示的业务逻辑、数据场景、界面呈现和讲述方式先对齐。

第一阶段交付物调整为：围绕 8 个核心演示场景，各产出一张高信息密度静态图和对应页面 prompt。静态图可以来自 Web 原型截图、AI Studio 生成图、设计稿或前端页面截屏，但验收重点不是技术实现方式，而是该图能否讲清楚监管价值。

## 2. 演示主线

演示主线围绕能源部一期关注的四类能力展开：

```text
实时感知 -> 异常预警 -> 自动归因 -> 全链留痕 -> 监管报告
```

这条主线要服务能源部总部监管视角，而不是企业内部设备运维视角。设备、管线、传感器和计量装置可以出现，但它们的作用是支撑能源部识别企业和作业区层面的监管风险。

## 3. 第一阶段目标

第一阶段目标是按飞书《1. 业务及场景设计》文档中的 8 个子场景，逐个完成静态图、页面生成 prompt 和评审说明卡，并在最近一次评审前完成双方口径对齐。

每个场景必须回答：

1. 这个页面解决什么监管问题？
2. 页面中间最重要的视觉对象是什么？
3. 页面使用哪些数据，其中哪些是真实公开资料，哪些是 mock？
4. 用户点击哪里，会看到什么？
5. 演示时这页要讲哪 3 句话？

## 4. 8 个演示场景

以下 8 个场景以飞书《1. 业务及场景设计》嵌入表格为准。后续评审按 1.1 到 4.2 的顺序逐个过，不再按 P0/P1/P2 砍范围。

### 4.1 1.1 全国能源设施实时态势图

定位：实时感知入口页。

页面应以哈萨克斯坦全国地图为底图，默认展示油气态势，并支持切换到电力和供暖。每个图标背后是一个或一组物联设备，包括电表、传感器、控制器、SCADA 节点等，按 15 分钟粒度刷新。本场景从头设计，不沿用现有首页 demo。

全国层级的重点不是展示某一个油田的细节，而是展示州级工业区、管输/港口、炼化/石化、出口节点之间的产业链走向和状态差异。阿克套/曼吉斯套应作为重点下钻区域出现，但全国入口必须同时覆盖阿特劳、西哈州、阿克托别、克孜勒奥尔达、巴甫洛达尔、奇姆肯特等关键州级链路。

关键呈现：

- 地图上的州级状态差异、能源节点、产业链路和状态色。
- 15 分钟级数据刷新提示。
- 顶部 KPI：在线设备数、离线设备数、异常数、数据延迟。
- 左侧筛选器：默认选中油气，可切换电力、供暖；同时支持地区、企业、设备等级筛选。
- 右侧实时事件流：最近 15 分钟接入数据的关键事件。
- 底部时间轴：回溯过去 24 小时态势。

### 4.2 1.2 计量装置在线档案

定位：实时感知的对象详情页。

页面应说明每一台纳入登记册并通过计量检定的认证测量装置都有一张“在线身份证”，展示设备静态属性和实时动态属性。

关键呈现：

- 设备或设施画像卡片。
- 设备编号、型号、制造商、认证证书、安装位置、所属企业。
- 当前读数、通信状态、采集时间、校准有效期。
- 最近 24 小时通信心跳图。
- 计量检定记录和证书图片。

### 4.3 2.1 时序大模型异常检测中心

定位：异常预警之一，偏物理链路和设施网络。

页面应展示能源行业时序大模型对接入设备和链路的 15 分钟级时序数据进行连续分析，并同时使用三层算法：简单阈值规则、统计趋势、时序大模型预测。

关键呈现：

- 中部主图展示链路、节点和异常段。
- 曲线展示实际值、预测值、预测带和异常区间。
- 右侧 AI 解释面板说明风险原因、置信度和建议核查动作。
- 底部待处置异常清单显示异常类型、识别算法和分级。
- 重点讲清“从事后处置提前到事前预警”。

### 4.4 2.2 跨系统数据交叉验证

定位：异常预警之二，偏跨部门申报逻辑核验。

页面应展示同一企业向能源部、税务部、海关、统计局等部门报送的数据之间是否存在逻辑矛盾，并通过多个专业 Agent 做一致性比对。

关键呈现：

- 顶部按企业搜索和筛选。
- 企业为中心节点，周围展开自报送、电表实测、税务、海关、统计等数据源。
- 连线标注逻辑公式和当前偏差率。
- 下方矛盾事件列表展示矛盾对象、比例、金额和 AI 可疑度。
- 右侧给出核查、约谈、立案等建议动作。

### 4.5 3.1 多 Agent 协同归因分析

定位：异常识别后的解释和研判页。

页面应在异常被识别后，启动多 Agent 归因工作流。审批 Agent、报送 Agent、检查 Agent、处罚 Agent、关联 Agent 分别调取不同数据，由主审 Agent 汇总归因结论和证据链。

关键呈现：

- 顶部锁定一个待归因异常事件。
- 中央展示 Agent 协同流程图。
- 每个 Agent 显示调取了什么数据、推理了什么、输出了什么。
- 右侧列出 3 到 5 个候选原因，并按概率排序。
- 底部证据链表可追溯到原始数据、文档或系统。

### 4.6 3.2 监管知识图谱探索

定位：通用监管分析能力展示页。

页面应把能源行业的法人、个体工商户、设施、设备、计量装置、人员、审批事项、违规事件建成知识图谱节点，并展示节点之间的持股、运营、关联、违规、整改等关系。

关键呈现：

- 中央力导向关系图。
- 节点类型包括企业、设施、人员、事件、装置。
- 边的颜色和粗细表达关系类型与强度。
- 顶部搜索栏定位节点。
- 右侧节点详情展示属性、关联节点和 AI 洞察。
- 底部高亮风险传染路径。

### 4.7 4.1 监管事件全生命周期审计链

定位：人在回路和可审计治理页。

页面应展示一个监管事件从 AI 识别异常、AI 归因、操作员复核、AI 建议处置、人工决策、工单流转、处置执行、整改复核到结案归档的完整留痕。

关键呈现：

- 顶部锁定监管事件 ID。
- 中央时间轴展示事件全生命周期。
- 节点标注时间戳、责任人、动作类型。
- 颜色区分 AI 自动、人工确认、人工修改 AI 建议、人工驳回 AI。
- 右侧合规性卡片评估 HITL、签批链和审计完整性。

### 4.8 4.2 分级监管报告自动生成

定位：监管报告生成和发布收束页。

页面应对应监管条款 46，根据预设模板和实时数据自动生成日报、周报、月报、紧急专报、部长简报，并记录引用数据、时间快照、AI 措辞和人工修改。

关键呈现：

- 左侧报告模板库。
- 中央报告预览区，AI 生成段落用淡色标记。
- 右侧数据快照面板展示报告所用数据版本。
- 底部流转区展示起草、AI 生成、编辑、签批、发布。
- 每份报告附 AI 生成声明和人工签批记录。

## 5. 协作方式

### 5.1 角色分工

用户负责业务判断和最终取舍：

- 判断哪些场景最能打动能源部。
- 判断 mock 数据是否合理。
- 判断界面讲述顺序是否符合演示节奏。
- 决定哪些内容进入最终演示。

Codex 负责结构化和落地：

- 把会议讨论整理成 PRD、任务卡和页面 prompt。
- 把每个场景拆成页面结构、数据字段、交互说明和验收标准。
- 根据已有前端工程实现可点击页面或静态页面。
- 检查页面是否符合 B 端监管系统风格。

另一位协作者负责发散和画面补全：

- 结合业务想象力补充场景细节。
- 产出静态图或页面 prompt。
- 提供素材、设计规范和已有页面截图。
- 与用户共同判断哪些画面值得合并。

### 5.2 协作节奏

第一轮：先对齐 8 个场景名称和顺序。

第二轮：每个场景补齐一页说明卡，说明业务价值、页面结构、数据、交互和讲述词。

第三轮：各自生成静态图或页面 prompt，不互相干预创作。

第四轮：合并两边最好的画面和 prompt，形成统一页面方案。

第五轮：由一个人集中落工程，避免多人同时改页面导致风格和组件割裂。

### 5.3 任务卡规则

每次派给 Codex、AI Studio、Gemini 或其他模型前，都先在 `docs/06-handoff/` 放任务卡。任务卡必须写清：

- 输入文件。
- 输出文件。
- 允许修改范围。
- 禁止修改范围。
- 场景目标。
- 页面必须出现的信息。
- 验收标准。

## 6. 场景推进顺序

8 个场景全部在第一阶段范围内。推进顺序严格按照飞书文档中的编号逐个过：

1. 1.1 全国能源设施实时态势图。
2. 1.2 计量装置在线档案。
3. 2.1 时序大模型异常检测中心。
4. 2.2 跨系统数据交叉验证。
5. 3.1 多 Agent 协同归因分析。
6. 3.2 监管知识图谱探索。
7. 4.1 监管事件全生命周期审计链。
8. 4.2 分级监管报告自动生成。

每过一个场景，都先确认说明卡，再写页面 prompt，再决定静态图如何生成。

## 7. 风险边界

- 不把 mock 数据包装成真实监管数据。
- 不把 AI 初判写成已确认违规。
- 不从企业设备运维视角替代能源部总部监管视角。
- 不为了炫技堆叠 dashboard、图谱和 agent 流程。
- 不让多人同时拆工程页面，先统一业务和画面，再集中实现。
